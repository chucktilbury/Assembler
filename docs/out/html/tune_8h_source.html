142 /*
143  * The C99 standard mandates that the operator `##' must yield a single,
144  * valid token, lest undefined behaviour befall upon thy head. Hence,
145  * for instance, `+ ## +=' is forbidden, because `++=' is not a valid
146  * token (although it is a valid list of two tokens, `++' and `=').
147  * However, ucpp only emits a warning for such sin, and unmerges the
148  * tokens (thus emitting `+' then `+=' for that example). When ucpp
149  * produces text output, those two tokens will be separated by a space
150  * character so that the basic rule of text output is preserved: when
151  * parsed again, text output yields the exact same stream of tokens.
152  * That extra space is virtual: it does not count as a true whitespace
153  * token for stringization.
154  *
155  * However, it might be desirable, for some uses other than preprocessing
156  * C source code, not to emit that extra space at all. To make ucpp behave
157  * that way, define the DSHARP_TOKEN_MERGE macro. Please note that this
158  * can trigger spurious token merging. For instance, with that macro
159  * activated, `+ ## +=' will be output as `++=' which, if preprocessed
160  * again, will read as `++' followed by `='.
161  *
162  * All this is irrelevant to lexer mode; and trying to merge incompatible
163  * tokens is a shooting offence, anyway.
164  */
165 /* #define DSHARP_TOKEN_MERGE */

187 /*
188  * Arithmetic code for evaluation of #if expressions. Evaluation
189  * uses either a native machine type, or an emulated two's complement
190  * type. Division by 0 and overflow on division are considered as errors
191  * and reported as such. If ARITHMETIC_CHECKS is defined, all other
192  * operations that imply undefined or implementation-defined behaviour
193  * are reported as warnings but otherwise performed nonetheless.
194  *
195  * For native type evaluation, the following macros should be defined:
196  * NATIVE_SIGNED the native signed type
197  * NATIVE_UNSIGNED the native corresponding unsigned type
198  * NATIVE_UNSIGNED_BITS the native unsigned type width, in bits
199  * NATIVE_SIGNED_MIN the native signed type minimum value
200  * NATIVE_SIGNED_MAX the native signed type maximum value
201  *
202  * The code in the arith.c file performs some tricky detection
203  * operations on the native type representation and possible existence
204  * of a trap representation. These operations assume a C99-compliant
205  * compiler; on a C90-only compiler, the operations are valid but may
206  * yield incorrect results. You may force those settings with some
207  * more macros: see the comments in arith.c (look for "ARCH_DEFINED").
208  * Remember that this is mostly a non-issue, unless you are building
209  * ucpp with a pre-C99 cross-compiler and either the host or target
210  * architecture uses a non-two's complement representation of signed
211  * integers. Such a combination is pretty rare nowadays, so the best
212  * you can do is forgetting completely this paragraph and live in peace.
213  *
214  *
215  * If you do not have a handy native type (for instance, you compile ucpp
216  * with a C90 compiler which lacks the "long long" type, or you compile
217  * ucpp for a cross-compiler which should support an evaluation integer
218  * type of a size that is not available on the host machine), you may use
219  * a simulated type. The type uses two's complement representation and
220  * may have any width from 2 bits to twice the underlying native type
221  * width, inclusive (odd widths are allowed). To use an emulated type,
222  * make sure that NATIVE_SIGNED is not defined, and define the following
223  * macros:
224  * SIMUL_ARITH_SUBTYPE the native underlying type to use
225  * SIMUL_SUBTYPE_BITS the native underlying type width
226  * SIMUL_NUMBITS the emulated type width
227  *
228  * Undefined and implementation-defined behaviours are warned upon, if
229  * ARITHMETIC_CHECKS is defined. Results are truncated to the type
230  * width; shift count for the << and >> operators is reduced modulo the
231  * emulatd type width; right shifting of a signed negative value performs
232  * sign extension (the result is left-padded with bits set to 1).
233  */
234
